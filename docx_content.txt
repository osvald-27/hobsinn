PART ONE — ANALYSIS PHASE
1. Executive Summary
HOBSINN is the operational platform of HOBS Innovations. It is a distributed waste management and community health coordination system designed for urban and peri-urban communities in Cameroon. The name HOBSINN reflects the startup brand identity while the platform tracks two strategic tracks — environmental coordination and community health.
The platform closes three immediate citizen-facing failures: households have no structured mechanism to schedule waste pickup, large unmanaged dumps accumulate because reporting them produces no coordinated financial or logistical response, and citizens who see dirty environments have no formal channel that connects their observation to a cleanup action.
The system is built on a five-tier distributed architecture using Java and Kotlin microservices, Apache Kafka for asynchronous event transport, PostgreSQL with PostGIS for geospatial data, MongoDB for analytics, and an Android mobile app. It is containerised with Docker and deployable to Kubernetes. The architecture is designed to absorb three future capability tracks: drone-based autonomous waste monitoring (UC4), clinical environmental sensor networks (UC5), and a recycling digital pipeline (UC6).
Every actor in the system — from the home user to the ambassador — is treated as a class hierarchy: every user is first a home user, with pickup users and ambassadors inheriting and extending those base capabilities. This inheritance principle simplifies design, reduces duplication, and makes the system intuitive.
Survey data from 27 respondents confirms that 100% recognise the link between poor sanitation and disease, only 19% consider their community clean, and 57.1% identify regular waste collection as the single highest-impact improvement. Two specific technical challenges — poor network connectivity and low digital literacy — are addressed with a customer service fallback (system admin creates accounts on behalf of users) and a planned offline/free-mode in the future.

6. User Behaviour Insights
The barrier to participation is structural, not motivational. 100% recognise the danger yet only 23.8% always act. HOBSINN is the coordination mechanism that converts existing motivation into consistent behaviour.
47.6% will join a cleanup if notified in advance. This is sufficient to run effective campaigns in Buea without requiring broad behaviour change.
Social media at 42.9% in a 95.2% aged 18–24 user base means the platform's engagement strategy must be social-native. Push notifications alone are insufficient — certificates and campaign completions must be shareable.
Malaria at 52.4% as the most cited disease implies stagnant water from waste is the community's primary concern. Dump clearance near drainage and water sources must receive elevated priority scores in the MatchingEngine.
Health professionals respond to environmental data with direct action at 83.3%. They are platform advocates, not passive recipients.
Buea at 84.2% of respondents confirms it as the correct pilot city and shapes initial ambassador and pickup user recruitment geography.

10. Internal Developer Audit Questions
10.1 Architecture
Why five tiers? The separation between data acquisition and backend prevents future drone or sensor data pipelines from mixing with business logic. Adding UC4 slots into the acquisition tier with no changes to existing services.
Why a monorepo? Shared event schemas, domain objects, and build tooling are version-controlled together. Services that share the common-events library always use the same event contracts.
Why Kafka over a simpler queue? The TIES prototype already uses Kafka. UC4 integration requires subscribing to a Kafka topic. Using Kafka from day one avoids a migration step.
Why the transactional outbox pattern? Without it, a business transaction can commit to the database while the Kafka publish fails. The outbox guarantees at-least-once event delivery with no lost events.
Why does every actor inherit from home user? Every person using the system has a home. The home user is the base class — pickup users and ambassadors extend it. This eliminates duplication and makes permission escalation explicit.
10.2 Data and Privacy
How is patient data handled in UC5? Clinical data is ingested only as anonymised aggregates — disease incidence counts per geographic zone per time period. No individual patient records are ever stored or processed by HOBSINN.
What data is retained and for how long? Operational data: 24 months then archived. Geospatial environmental data: indefinitely for analytics. User identity: retained while account is active and 12 months post-deletion for audit.
10.3 Infrastructure — The 12-Factor Scale Design
All 12 scalability factors identified in the infrastructure brief are addressed in this system design:
Horizontal vs. Vertical Scaling: All application-tier services are stateless. Scale horizontally by adding instances behind the API Gateway load balancer. No vertical scaling required.
Load Balancing: API Gateway distributes traffic across all service instances. Kubernetes LoadBalancer service exposes only the gateway externally.
Stateless Architecture: Services hold no session state. All state lives in PostgreSQL, Kafka, or Redis. Any instance can handle any request at any time.
Database Sharding and Partitioning: PostgreSQL read replicas for analytics queries. TimescaleDB hypertables for sensor time-series data (UC5). Future horizontal sharding by geographic zone.
Caching Strategies: Redis 7.2-alpine for rate limiting and session token cache in the API Gateway. CDN for static assets and campaign images.
Asynchronous Processing: All cross-service communication via Kafka topics. Heavy operations (hotspot detection, reconciliation, certificate generation) are async and never block user-facing responses.
Microservices and Decomposition: Nine independent services — each deployable without restarting others. PaymentService can scale independently during campaign funding surges.
Database Replication: PostgreSQL read replica serves analytics-service and admin dashboard queries, isolating read traffic from transactional writes.
Auto-Scaling and Elasticity: Kubernetes HorizontalPodAutoscaler scales on CPU (70% target) and custom Kafka consumer lag metric. Cloud resources auto-scale via Terraform IaC.
Service Discovery: Kubernetes DNS-based service discovery for all internal service-to-service communication. No hardcoded service addresses.
Consistent Hashing: Redis cluster uses consistent hashing for distributed cache key distribution. Kafka partition assignment uses consistent hashing for consumer group load balancing.
Circuit Breakers: Spring Cloud Circuit Breaker wraps all inter-service HTTP calls and MoMo gateway calls. 503 responses with Retry-After headers instead of cascading failures.
10.4 AI and Analytics
What AI is needed for UC1, UC2, and UC3? The MatchingEngine uses a weighted scoring algorithm, not ML. The HotspotDetector uses DBSCAN spatial clustering. The optional bag volume estimator uses a computer vision model. Analytics metrics are computed aggregation queries.
How do we complete implementation in under 24 hours? Solo relay implementation using Claude Code instances. Each instance receives a precise handoff document specifying exact files completed, current state, and exact next tasks with no ambiguity.
How do we keep up with AI-generated code and ensure reliability? All AI-generated code passes through CI/CD with automated tests. DevOps monitoring dashboards provide real-time reliability signals.

PART TWO — DESIGN PHASE
15. Architectural Philosophy and System Overview
HOBSINN is not a reporting application. It is a coordination platform that closes the loop from problem detection through resource mobilisation, service execution, and verified resolution. The architecture reflects this completeness.
Four principles govern every design decision. Deployability now: the system must function today with existing infrastructure, existing payment methods, and existing operators. Extensibility by design: every interface that UC4, UC5, and UC6 will consume is defined now, even though those consumers do not yet exist. Strict separation of concerns: each tier, each service, and each class has one reason to exist. Trust through transparency: every financial transaction, every status transition, and every campaign milestone must be auditable by the involved parties.
The actor inheritance model is a key architectural principle: every user first IS a home user. The pickup user extends the home user. The ambassador extends the home user. This is enforced at the data model, API, and UI layers. A pickup user cannot see his or her own pickup requests as a client.
15.1 System Boundary
HOBSINN coordinates service delivery between home users, pickup users, ambassadors, and payment infrastructure. It does not physically collect waste, process medical waste, operate drones in the current phase, or manage industrial waste. These exclusions are deliberate boundary decisions that keep the system focused and legally clear.

19.4 Transactional Outbox Pattern
All services that publish Kafka events use the transactional outbox pattern. When a business operation commits to PostgreSQL, the event is written to the outbox_events table in the same database transaction — not sent directly to Kafka. A background OutboxPublisher polls the table every 2 seconds, sends pending events to Kafka, and marks them PUBLISHED. Maximum retry count is 5 before the event is marked FAILED.
19.5 Notification Service
NotificationService is a Kafka consumer subscribed to all event topics. It resolves the delivery channel per recipient (FCM push if device token exists, SMS via Twilio or local telecom gateway as fallback), renders the correct language template (English or French), and dispatches asynchronously. Client-side calendar caching enables notifications to surface offline. A notification failure never blocks a business transaction.

21.4 CostEstimationModule — Dump Campaigns
Stateless. No constructor dependencies. No database access. Pure function.
estimatedAmount = amountPerMeterCubed × volumeM3
platformFee = 5000 XAF + (10% × estimatedAmount if estimatedAmount > 25,000 XAF)
totalCost = estimatedAmount + platformFee
Ambassador earns 10% of estimatedAmount. The 90% remainder is split among pickup users per ambassador-configured percentages.

22.2 HotspotDetector
Algorithm: DBSCAN (Density-Based Spatial Clustering of Applications with Noise). Distance metric: Haversine formula. Parameters: epsilon = 500 metres, minSamples = 3 (both configurable in reporting_config table). Time window: 30 days. Input: List<ReportCoordinate>. Output: List<DetectedHotspot> with centroid, radius, dominant severity, and member report IDs.
22.3 BagVolumeEstimator
A TensorFlow Lite model running on-device in the Android app. Input: JPEG image of a waste bag taken by the home user. Output: estimated volume in litres. The estimated volume is sent to the backend where pricing is calculated per litre. Model inference runs on-device to function without network connectivity. Falls back to manual bag count entry if model fails or image quality is too low.
22.4 Job Scoring System for Pickup Users
After every completed job, a composite score is computed: (0.40 × punctualityScore) + (0.30 × jobCompletedStatus) + (0.30 × customerRating). Punctuality score starts at 100% and degrades by 25% per 5 minutes late. If pickup arrives within 20-minute grace window, punctuality is calculated at that point. Customer rating is 1–6 stars, normalised to 0–1. The composite score for each job is blended into the pickup user's running reliability score using an exponential moving average. The reliability score is then converted to a 1–6 star display rating.

Primary operational data: PostgreSQL 15 + PostGIS 3.4. ACID compliance, native geospatial types (GEOGRAPHY with SRID 4326), JSON config storage.

Schema migrations: Flyway versioned SQL files. V1__initial_schema.sql. Every schema change in a new numbered migration file. No manual schema changes in any environment.

27. Use Case Specifications
UC-001: Schedule Special Call Waste Pickup
Field
Detail
Actors
Primary: Home User. Secondary: Pickup User, MTN MoMo Gateway, Notification Service, MatchingEngine.
Preconditions
User has HOME_USER role. User enables GPS. User has active MoMo account.
Main Flow
1. User enables location sharing. 2. Enters bag count and optional photos for AI volume estimation. 3. Selects datetime (> 1 hour away). 4. System calculates total cost and displays breakdown. 5. Request published. MatchingEngine biases top 3 pickup users. 6. Pickup user accepts from notification or Special Calls list → MATCHED. 7. Home user confirms payment via MoMo ≥ 1 hour before agreed time → CONFIRMED. 8. Home user acknowledges arrival → IN_PROGRESS. 9. Pickup user submits waste photo → COMPLETED. Escrow released.
Alternative A1
No pickup user available within 25km → User notified, offered to reschedule.
Alternative A2
Pickup user accepts then resigns > 1 hour before agreed time → request returns to PENDING with no penalty.
Alternative A3
Payment not confirmed by 1-hour deadline → CANCELLED. Home user badge degraded considerably.
Alternative A4
Pickup user resigns within 1-hour danger zone → 0% score on that job, badge degradation. Request → PENDING (marked invalid for imminent reschedule).
Exception E1
Pickup user late > 20 minutes → triggerReassignment(). REASSIGNED. Home user notified.
Exception E2
Home user cancels after CONFIRMED but > 1 hour before → full refund, no penalty (platform keeps fee).
Exception E3
Home user cancels within 1-hour danger zone → pickup user paid 50% of agreed amount. Platform keeps its charges. Home user badge mildly degraded.
Postconditions
PickupRequest status=COMPLETED. Transaction status=RELEASED. Pickup user rating score updated.
UC-002: General Pickup Slot Registration
Field
Detail
Actors
Primary: Home User, Pickup User. Secondary: Ambassador (capacity verification), Payment Service.
Preconditions
General slots created by system admin. Pickup user vehicle verified by ambassador.
Main Flow
1. Home user selects a general slot (day, time, community). Inputs estimated bag count for the week. 2. Pickup user subscribes to monthly slot coverage. System calculates aggregated capacity from all registered home users. 3. Pickup user price for the slot adjusts: increases if aggregate capacity is high, decreases if low. 4. Home user pays pickup user hand-to-hand on pickup day. 5. Both events recorded in calendar. Pickup user Ongoing shows upcoming general pickups.
Alternative A1
Pickup user capacity insufficient for aggregated demand → System splits slot or seeks additional pickup user.
Postconditions
GeneralSlotSubscription recorded. GeneralSlotRegistration recorded. Calendar events created for both parties.
UC-003A: Major Dump Reporting and Funded Cleanup
Field
Detail
Actors
Primary: Home User (reporter), Ambassador, Pickup Users (1–3). Secondary: Contributing Home Users, MTN MoMo Gateway.
Preconditions
User has HOME_USER role. User GPS at dump location.
Main Flow
1. Home user submits dump report with GPS, description, photos. 2. Ambassador receives and validates: inputs area, height, contact → PENDING. 3. Job posted for pickup registration (max 3 pickup users, 3 days window). 4. Ambassador contacts pickup users, inputs payment percentages → CONFIGURED. 5. Cost calculated. Campaign published → PUBLISHED. 6. Home users contribute via MoMo. Leaderboard updated. Overflow is platform profit. 7. Campaign reaches target → FUNDED. 8. Ambassador starts cleanup within 10 days → IN_PROGRESS. 9. Completion: ambassador submits after photo, celebration text, final percentage adjustments → COMPLETED. Escrow released. Notification with shareable link posted to all.
Alternative A1
3 days pass without CONFIGURED → campaign CANCELLED. Coordination failure flagged.
Alternative A2
10 days pass without start → ambassador loses 10% fee, receives badge degradation. System admin notified.
Postconditions
DumpCampaign status=COMPLETED. All payment percentages disbursed. Ambassador badge upgraded significantly. Campaign completion shareable to social media.
UC-003B: Dirty Environment Volunteer Cleanup
Field
Detail
Actors
Primary: Home User (reporter), Ambassador, Volunteer Home Users.
Preconditions
User has HOME_USER role. User GPS at location.
Main Flow
1. Home user submits dirty environment report with GPS, severity, optional photo. 2. Ambassador validates: inputs photo, area estimate, contact → OPEN. 3. 29-day volunteer window. Volunteers input real name and commitment date. 4. After 29 days: ambassador acknowledges within 1 day → IN_SESSION. Community votes on session date (within 7 days). 5. Cleanup day: ambassador opens QR scan. Scans each attendee → system marks ATTENDED. 6. Ambassador acknowledges completion. Certificates generated and emailed to attended volunteers.
Postconditions
CleanupCampaign status=COMPLETED. Certificates issued with volunteer names, date, location, VOLUNTEER_PROGRAM type.

32. Scalability, Reliability and Fault Tolerance
32.1 Scalability Strategy — All 12 Factors Applied
Factor
How HOBSINN Applies It
1. Horizontal vs. Vertical Scaling
All application-tier services are stateless by design. Scale by adding instances behind the API Gateway load balancer. Kubernetes HPA auto-scales on CPU (70% target) and Kafka consumer lag.
2. Load Balancing
API Gateway distributes traffic across all service instances using round-robin by default. Kubernetes LoadBalancer service exposes only the gateway externally.
3. Stateless Architecture
Services hold no session state. All state lives in PostgreSQL, Kafka, or Redis. Any instance can handle any request at any time.
4. Database Sharding and Partitioning
PostgreSQL read replica for analytics-service. TimescaleDB hypertables for sensor time-series data (UC5). Future geographic zone sharding as user base grows.
5. Caching Strategies
Redis 7.2-alpine caches JWT session tokens and API Gateway rate limiting counters. Client-side Android calendar cache enables offline access. CDN for static campaign images and certificates.
6. Asynchronous Processing
All cross-service communication via Kafka. Hotspot detection, certificate generation, leaderboard updates, and email dispatch are async. Heavy operations never block user-facing API responses.
7. Microservices and Decomposition
Nine independent services. CampaignService and NotificationService scale independently during campaign surges. PaymentService scales independently during MoMo payment peaks.
8. Database Replication
PostgreSQL read replica serves analytics-service and admin dashboard read queries, isolating read traffic from transactional writes on the primary.
9. Auto-Scaling and Elasticity
Kubernetes HorizontalPodAutoscaler scales on CPU and Kafka consumer lag. Terraform IaC provisions cloud resources (VPC, RDS, MSK, EKS) that auto-scale.
10. Service Discovery
Kubernetes DNS-based service discovery for all internal communication. No hardcoded service addresses. Services locate each other by Kubernetes service name.
11. Consistent Hashing
Redis cluster uses consistent hashing for distributed cache key distribution. Kafka partition assignment uses consistent hashing for consumer group load balancing.
12. Circuit Breakers
Spring Cloud Circuit Breaker wraps all inter-service HTTP calls and MoMo gateway calls. Returns 503 with Retry-After header instead of queuing indefinitely or cascading failures.
32.2 Reliability Design
Transactional outbox pattern guarantees at-least-once Kafka event delivery. Business transactions and event creation commit atomically to PostgreSQL.
Payment idempotency key (sourceEntityType:sourceEntityId) prevents duplicate charges on retry. The gateway_transaction_id unique constraint prevents duplicate webhook processing.
Pickup user late detection runs on a 1-minute polling cycle for CONFIRMED requests approaching their agreed datetime. At 20 minutes late: auto-reassignment.
Campaign deadline enforcement runs daily. Queries PENDING_PICKUP campaigns past 3-day window, FUNDED campaigns past 10-day cleanup window. Auto-transitions and notifications dispatched.
32.3 Fault Tolerance Design
If NotificationService is unavailable, all business transactions still complete. Notifications are queued as outbox events and dispatched when the service recovers.
If MatchingEngine returns no results, SchedulingService returns a clear no-pickup-available response and logs the failed match for analytics.
If MoMo gateway is unreachable, PaymentService queues the initiation request. Reconciliation job running every 15 minutes retries gateway verification. After 5 failed retries, transaction marked FAILED and slot released.
Client-side calendar caching in the Android app ensures scheduled events surface as notifications even without an active data connection.
Kafka consumer failures produce dead letter queue events after configured max retry attempts. Dead letter events are flagged in Grafana alerts for administrator investigation.

34. AI Modelling Detail
34.1 MatchingEngine — Scoring Algorithm
Input features per pickup user candidate: distance_km (Double), reliability_score (Double, 0–1, derived from exponential moving average of composite job scores), capacity_litres (Int), current_accepted_jobs_in_window (Int). Configuration: matching_config table — w_distance, w_reliability, w_capacity, w_workload (all Double, sum must equal 1.0).
Normalisation: distanceScore = 1 - (distanceKm / maxDistanceKm). workloadScore = 1 - (currentAcceptedJobs / maxAcceptedJobs). capacityScore = 1 - |1 - (requiredVolumeL / vehicleCapacityL)| clamped to [0,1]. Final compositeScore clamped to [0,1].
34.2 HotspotDetector — DBSCAN
Algorithm: DBSCAN. Distance metric: Haversine (Earth radius 6371 km). Parameters in reporting_config table: epsilon_meters (default 500), min_samples (default 3), time_window_days (default 30). Core point: report with ≥ min_samples neighbours within epsilon_meters. Output per cluster: centroid (mean lat/lng), radiusMeters, reportIds list, dominantSeverity (mode across cluster members).
34.3 BagVolumeEstimator — TFLite Vision Module
Model type: MobileNet V2 fine-tuned regression model. Input: 224×224 RGB image of a single waste bag. Output: estimated volume in litres (float). Inference runs on-device (Android TFLite runtime) with no network required. Training data: labelled images of waste bags in Cameroonian household contexts with known volumes. Fallback: if model confidence < 0.6 or image quality insufficient, prompt user to enter bag count manually.
34.4 Job Scoring System — Specification
For each completed or evaluated job:
punctualityScore = max(0, 1 - (minutesLate × 0.25)). Clamped to [0, 1].
completionStatusScore = 1.0 if COMPLETED, 0.0 if REASSIGNED or CANCELLED by pickup user.
customerRatingScore = (customerRating - 1) / 5. Maps 1-star to 0.0, 6-star to 1.0.
jobCompositeScore = (0.40 × punctualityScore) + (0.30 × completionStatusScore) + (0.30 × customerRatingScore).
Reliability score update (exponential moving average, alpha = 0.2):
newReliabilityScore = (alpha × jobCompositeScore) + ((1 - alpha) × previousReliabilityScore).
Star rating display = ceil(newReliabilityScore × 6). Range: 1–6 stars.
34.5 WasteSpreadPredictor — Future UC4
Model type: Spatial-temporal regression. Input features: drone observation time series, wind direction/speed from weather API, rainfall probability, terrain DEM, drainage network proximity. Training data requirement: minimum 6 months of paired observations. Output: probability surface map as GeoTIFF, top-5 highest-risk zones for next 7 days.
34.6 ClinicalCorrelationEngine — Future UC5
Model type: Multivariate time-series correlation with lagged variables. Target: weekly disease incidence rate per geographic zone. Input features: PM2.5, CO, water turbidity 7-day rolling averages, waste density. Minimum Pearson r = 0.60 before advisory is issued. Output: risk score per zone, advisory level (Normal / Elevated / Warning / Alert), 95% confidence interval.

35.2 Solo Relay Implementation Protocol
Because the hackathon implementation is performed by relay instances of Claude Code, each instance MUST produce a handoff document at the end of its session containing:
Exact list of files created or modified with their full paths.
Current state of the system: which services compile, which endpoints pass Postman tests, what is still broken.
Exact next task with file path, class name, method name, and all context needed to continue.
Any environment variables, secrets, or infrastructure state that the next instance needs to know.
Any blockers or decisions deferred to the next instance with enough context to resolve them.
This handoff document is the single source of truth for the relay. Without it, the next instance starts from scratch.
35.3 Coding Standards
Every public service method has a KDoc comment describing inputs, outputs, exceptions, and side effects.
No hardcoded values anywhere. All configurable parameters in application.properties or environment variables. No credentials in code.
Unit tests required for: all MatchingEngine scoring logic, all state machine transition validations, all CostEstimationModule computations, all payment idempotency checks, BagVolumeEstimator fallback logic, job scoring computation.
Integration tests use Testcontainers: real PostgreSQL + PostGIS, real Kafka, real Redis.
Every database schema change is a new numbered Flyway migration file. No manual ALTER TABLE in any environment.
Git commits follow conventional commit format: feat(uc1): add special call creation endpoint
Pull requests require one reviewer approval and a green CI pipeline before merge.
